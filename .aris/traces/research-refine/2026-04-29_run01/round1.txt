# Research Proposal: Protocol Beats Structure — A Diagnostic Study of Agentic GRPO for Text-to-SQL

## Problem Anchor

- **Bottom-line problem**: Multi-turn ReAct GRPO on Text-to-SQL succeeds modestly at 7B (+2 pp Spider-dev EX_att/EX_all) but fails at 3B; a structural-advantage prior (SKE-RL) fails to help and worsens with more rollouts. We do not know what the +2 pp actually changed inside the agent, and we cannot explain why a natural structural prior fails.
- **Must-solve bottleneck**: Diagnostic gap. The literature reports aggregate EX gains without decomposing protocol-compliance vs SQL-reasoning contributions, so neither positive (Dense GRPO) nor negative (SKE) results can be interpreted mechanistically.
- **Non-goals**: SOTA on Spider/BIRD; new method that fixes everything; scaling-law claim across model sizes; a toolkit-as-lead paper.
- **Constraints**: 1× Quadro RTX 6000 24 GB, ~3 weeks GPU + 1 week writing; SKE-RL stack + per-class diagnostics + full Spider-dev eval JSONs already on disk; workshop-tier deadline driven.
- **Success condition**: A reviewer would say the paper changes how they read agentic-SQL-RL leaderboards (because of the protocol decomposition) AND the SKE negative result has a single, convincing, measured mechanism (not "we tried it and it didn't work").

## Technical Gap

Recent agentic Text-to-SQL RL work (HES-SQL, MTIR-SQL, MARS-SQL, Graph-Reward-SQL) reports +N pp EX_all gains and treats those gains as evidence of improved SQL reasoning under structural / execution / multi-agent supervision. **No paper in this line decomposes the gain into "the model produced more valid trajectories" vs "the model produced semantically better SQL".** The two are systematically conflated under EX_all because EX_all = (attempted-rate × EX_att), and aggregate metrics give attempted-rate gains the same weight as semantic gains.

This conflation has direct consequences:
- Interpretation failure for existing skeleton-reward methods (HES-SQL).
- Negative-result opacity: SKE-RL fails but we cannot say which of {wrong abstraction, class fragmentation, optimization side-effect, variance-reduction failure} explains it.
- Naive bigger systems do not fix this — more rollouts / more rewards / multi-agent decomposition all report the same conflated metric.

The smallest adequate intervention is a **measurement protocol**, not a new training method. We instrument existing eval JSONs and one new diagnostic eval to split EX_all into a 4-cell taxonomy {format-fix / executable-but-wrong / executable-and-better-SQL / unchanged}, then map SKE's negative result onto that taxonomy to identify the mechanism.

## Method Thesis

- **One-sentence thesis**: Agentic GRPO on multi-turn Text-to-SQL improves aggregate accuracy primarily by aligning tool-use protocols (valid action format, valid SQL syntax, valid Answer extraction, repair-after-observation), not by improving SQL semantic reasoning — and this protocol-vs-semantic split is what makes the SKE-RL structural-advantage prior fail (it targets the layer below the actual bottleneck, and degrades with more rollouts because larger groups give the class-baseline more opportunity to wash out the few semantically-corrective trajectories).
- **Smallest adequate intervention**: A diagnostic measurement layer (lenient-extraction re-eval + win/loss transition table + repair-attribution counter). No new training. Existing 4 checkpoints (SFT, Dense75, SKE-G4-75, SKE-G8-75) become the data; the contribution is the lens through which they are read.
- **Why timely**: Foundation-model-era agentic RL all reports aggregate task-success gains. The protocol-vs-semantic conflation we identify on Text-to-SQL almost certainly applies to those settings. The paper offers a portable diagnostic, not a SQL-specific trick.

## Contribution Focus

- **Dominant**: Empirical diagnostic showing Dense agentic GRPO gain on Text-to-SQL is primarily protocol/tool-use alignment (lenient-extraction Δ ≪ strict-extraction Δ; transitions concentrated in {format_fix, observation_repair} not {semantic_repair}), and a portable protocol that surfaces this split in any agentic-RL evaluation.
- **Optional supporting**: Mechanism-grounded negative result for SKE-RL — fails because the bottleneck is at the protocol layer it cannot affect; G=8 monotonic degradation is the predicted consequence.
- **Explicit non-contributions**: No new RL method that wins; no scaling law; no SOTA; no claim about model "reasoning capacity"; no toolkit paper.

## Proposed Method

### Complexity Budget
- **Frozen / reused**: All 4 existing 7B checkpoints; existing eval JSONs with full trajectories; existing skeleton extractor; existing query-class tagger; existing aggregator. No new model training for dominant contribution.
- **New trainable components**: 0 for dominant contribution. For SKE mechanism: 1 new low-cost training run (Dense-G=8 control, ~1.2 run-eq) to disambiguate G-effect from SKE-effect.
- **Tempting additions intentionally not used**: Multi-seed averaging, BIRD GRPO, 1.5B/14B capacity points, v2 SKE, reward-side skeleton bonus large-scale comparison.

### System Overview

```
Existing eval JSONs (SFT, Dense75, SKE-G4-75, SKE-G8-75; full Spider-dev)
   │
   ├─ Diagnostic layer 1: STRICT vs LENIENT extraction
   │     re-extract any plausible final SQL from trajectory transcript
   │     re-execute, recompute EX_all_lenient
   │     ΔEX_lenient < ΔEX_strict  →  RL gain = protocol/format
   │
   ├─ Diagnostic layer 2: WIN/LOSS TRANSITION TAXONOMY (per-record)
   │     classify each query by SFT-vs-Dense outcome:
   │       format_fix        : SFT format/tool failed; Dense executes
   │       observation_repair: both attempt; Dense recovers via observation
   │       semantic_repair   : both produce executable SQL; Dense's correct (skeleton/tables/conditions changed)
   │       protocol_only     : trajectory shape identical; both right or both wrong
   │     report % of {SFT-wrong → Dense-right} in each bucket
   │
   ├─ Diagnostic layer 3: TURN-LEVEL REPAIR COUNTERS
   │     per-trajectory: # execute_sql actions, % first-action-failed, % final-correct-after-error
   │
   └─ SKE mechanism mapping
         apply same 3 layers to SKE-G4 / SKE-G8 vs Dense
         pick mechanism candidate whose signature matches:
           M1 variance-reduction failure  → measure adv variance per group; SKE doesn't reduce vs Dense
           M2 class fragmentation         → measure |distinct skeleton classes|/G; many singletons
           M3 wrong abstraction           → semantic_repair queries have HIGH within-skeleton-class outcome variance
           M4 optimization side-effect    → SKE shifts adv distribution toward zero/negative for exploratory rollouts
```

### Core Mechanism (= the diagnostic protocol)

Input: eval JSON with `question, gold_sql, pred_sql, trajectory, execution_match, db_id`.

Output: 4-row table per (RL vs SFT) pair —

| Bucket | Definition | Tells you |
|---|---|---|
| format_fix | SFT no extractable SQL OR malformed Action; RL trajectory executes | RL fixed protocol — *not reasoning* |
| observation_repair | Both attempt SQL; first fails; RL final correct via observation | RL improved error-recovery — *partial reasoning* |
| semantic_repair | Both executable; different skeleton/table-set; RL is correct | RL improved SQL semantics — *true reasoning gain* |
| protocol_only | Trajectory shape identical (or only SQL-equivalent edits); both executable | RL did not change the answer |

Plus headline number: `lenient_extraction_attribution = (ΔEX_strict - ΔEX_lenient) / ΔEX_strict ∈ [0, 1]`. 1.0 = all gain is protocol; 0.0 = all gain is semantic.

Why main novelty: No published agentic Text-to-SQL RL paper reports either measurement. Both are interpretive primitives, not engineering tricks. Generalize trivially to other tool-using agentic RL settings.

### Optional Supporting Component (= SKE failure mechanism)

Apply 3 diagnostic layers to SKE-G4-75 and SKE-G8-75 vs Dense75. Match signature against M1-M4. Each candidate has distinctive observable signature; data picks one. M3 (wrong abstraction) is *predicted by* the dominant thesis: if bottleneck is protocol, structural advantage prior cannot help, and class-baseline subtraction washes out rare semantic-repair signal — larger G makes this worse.

No contribution sprawl: M_* is *predicted by* the dominant thesis. Confirming it strengthens both, not adds parallel claim.

### Modern Primitive Usage
- **AST-canonical equivalence (sqlglot)**: in existing SKE-RL extractor + new `semantic_repair` detector to verify two SQLs differ in skeleton/tables, not just literals.
- **Lenient-extraction re-execution**: leverages real SQLite execution as inference-time-verifier — recovers what the agent meant to say despite protocol noise. Gap between tolerant-eval and strict-eval IS the protocol-attribution measurement.
- No new trained component for the dominant contribution. Leverage is observational.

### Integration
Diagnostic layer attaches AFTER an arbitrary agentic-RL pipeline. Reads eval JSONs only. No re-training, no model surgery.

### Training Plan
Dominant contribution: NONE. Support: one new GRPO run — Dense G=8 control, 75 steps, identical recipe to Dense G=4 except group size. Cost ~1.2 run-eq.

### Failure Modes and Diagnostics

| Failure | Detect | Fallback |
|---|---|---|
| Attribution ≈ 0 (Dense gain fully semantic) | Lenient ΔEX ≈ Strict ΔEX | Pivot story: "RL gives genuine reasoning lift; SKE still fails (M3 or M2)" — diagnostic still novel |
| Dense G=8 also degrades | Dense G=8 EX_att ≪ Dense G=4 | G-effect is generic; story shifts to "rollout-budget-vs-quality trade-off" — still publishable |
| Transition table dominated by `protocol_only` | Both pipelines produce same outcomes most queries | Δ effect is noise; report effect-size CI honestly |
| Lenient extraction unreliable | Lenient EX_all < Strict EX_all on SFT (sanity) | Tighten extractor to "any final-Answer-region SQL that parses with sqlglot" |
| Mechanism candidates ambiguous | Two M_i within 1 SD | Report both; do not over-claim single pick |

### Novelty and Elegance Argument

Closest published work: HES-SQL (skeleton REWARD), Graph-Reward-SQL (graph reward), MTIR-SQL (multi-turn GRPO), MARS-SQL (multi-agent ReAct), LearNAT (AST + DPO). All sit inside the EX_all conflation — propose intervention, show aggregate gain. We do NOT propose winning intervention. We propose a diagnostic that turns those existing aggregate gains into mechanistic claims, and use SKE-RL (designed-to-fail control) as the lens that makes the diagnostic visible.

Focused: One measurement layer. One supporting failure analysis predicted by main thesis. No new training for dominant contribution. Whole paper rotates around one question — "what does agentic SQL RL actually change?" — with exactly one new way to answer it.

Elegant: Diagnostic is one re-execution pass + one trajectory-classification table. Anyone with eval JSONs from any agentic Text-to-SQL RL paper can apply it tomorrow.

## Claim-Driven Validation Sketch

### Claim 1 (dominant): Dense agentic GRPO gain on Text-to-SQL is dominantly protocol/tool-use alignment

- **Min experiment**: Lenient-extraction re-eval of SFT vs Dense75 on full Spider-dev (1034); compute attribution. Win/loss transition table on the 200-query subset with rich logs. Cost ~0.4 run-eq for re-execution; 0 for table.
- **Baseline**: SFT vs Dense75; ΔEX_lenient vs ΔEX_strict.
- **Metric**: `attribution ∈ [0,1]`; bucket distribution of {SFT-wrong → Dense-right}.
- **Expected**: attribution ≥ 0.5; format_fix + observation_repair > 60%; semantic_repair < 25%.

### Claim 2 (supporting): SKE-RL fails because it targets SQL-structure layer while bottleneck is at protocol layer; G=8 degradation is predicted

- **Min experiment**: Apply diagnostic to SKE vs Dense at G=4 and G=8. Score 4 mechanism candidates from per-rollout SKE training logs (already captured) + new diagnostic. Plus Dense G=8 control (~1.2 run-eq).
- **Baseline**: 2×2 grid {Dense, SKE} × {G=4, G=8}.
- **Metric**: Per-mechanism signature score; M3 prediction "SKE wins on protocol_only / loses on semantic_repair, loss grows with G" is keystone.
- **Expected**: M3 wins; Dense G=8 ≥ Dense G=4 (so SKE degradation is structural to class baseline, not generic G effect).

## Experiment Handoff Inputs
- **Must-prove**: C1 protocol attribution; C2 SKE mechanism.
- **Must-run ablations**: Dense G=8 control; lenient-extraction sanity on SFT.
- **Critical metrics**: attribution, 4-bucket transitions, per-mechanism signatures.
- **Highest-risk assumptions**: (R1) lenient-extraction recovers SQL from format-failed SFT trajectories; (R2) `semantic_repair` bucket non-empty (≥10 records); (R3) Dense G=8 ≥ Dense G=4.
