# Research Proposal: Protocol Beats Structure — A Diagnostic Study of Agentic GRPO for Text-to-SQL (R1 revision)

## Problem Anchor
- **Bottom-line problem**: Multi-turn ReAct GRPO on Text-to-SQL succeeds modestly at 7B (+2 pp Spider-dev EX_att/EX_all) but fails at 3B; SKE-RL fails to help and worsens with more rollouts. We do not know what the +2 pp actually changed inside the agent, and we cannot explain why a natural structural prior fails.
- **Must-solve bottleneck**: Diagnostic gap. Literature reports aggregate EX gains without decomposing protocol-compliance vs SQL-reasoning contributions.
- **Non-goals**: SOTA; new method that wins; scaling-law claim; toolkit-as-lead.
- **Constraints**: 1× Quadro RTX 6000 24 GB, ~3 weeks GPU + 1 week writing.
- **Success condition**: Reviewer says paper changes how they read agentic-SQL-RL leaderboards AND SKE negative result has a single, measured, convincing mechanism.

## Technical Gap

Recent agentic Text-to-SQL RL work (HES-SQL, MTIR-SQL, MARS-SQL, Graph-Reward-SQL) reports +N pp EX_all and treats those gains as evidence of improved SQL reasoning. **No paper in this line decomposes the gain into protocol-validity vs SQL-semantics.** The two are systematically conflated under EX_all. Consequences: existing structural-reward methods (HES-SQL) cannot say whether their gain is structural or coincidental protocol cleanup; negative results like SKE-RL are uninterpretable; naive scaling reports the same conflated metric.

The smallest adequate intervention is a measurement protocol, not a new training method.

## Method Thesis

- **One-sentence thesis**: Agentic GRPO on multi-turn Text-to-SQL improves aggregate accuracy primarily by aligning tool-use protocols (valid action format, valid SQL, valid Answer extraction), not by improving SQL semantic reasoning — which explains why a structural-advantage prior (SKE-RL) cannot help: it operates on the layer below the actual bottleneck.
- **Smallest adequate intervention**: Deterministic 2-bucket classifier (`protocol_gain` / `semantic_gain` / `no_change`) + lenient-vs-strict re-execution + single attribution ratio. NO new training for dominant contribution.
- **Why timely**: Foundation-model-era agentic RL all reports aggregate task-success gains. Protocol-vs-semantic conflation is portable.

## Contribution Focus
- **Dominant**: Deterministic, portable diagnostic that decomposes any agentic Text-to-SQL RL gain into protocol-validity vs SQL-semantic components, with a single headline `attribution_ratio`. Applied to our 7B Dense GRPO, it shows the +2 pp gain is dominantly protocol.
- **Optional supporting**: SKE-RL fails because bottleneck is protocol-layer; same diagnostic confirms via 3 evidence requirements (E1+E2+E3).
- **Explicit non-contributions**: Mechanism grid M1-M4 (appendix only); new RL method; scaling law; SOTA; toolkit-as-lead.

```
                       │                    + last-action-fallback if no Answer,
                       │                    must sqlglot-parse
                       │      Re-execute both against db_path
                       ▼
                       Step 2: Per-record classifier (DETERMINISTIC)
                       Inputs: see classifier interface below
                       Outputs: bucket ∈ {protocol_gain, semantic_gain, no_change}
                                + flags (boolean set)
                       ▼
                       Step 3: Aggregate diagnostic table
                       For (RL vs SFT) pair:
                         attribution = (ΔEX_strict − ΔEX_lenient) / max(ΔEX_strict, ε)

                       ▼

```

### Core Mechanism

**Classifier interface** (deterministic):
```python
def classify(record) -> tuple[Bucket, Flags]:
    """
    Inputs (per-record dict):
      strict_pred_sql, lenient_pred_sql, gold_sql, all_sql_actions,
      execution_errors, final_answer_span, db_id

    Computed flags:
      strict_extract_ok    = strict_pred_sql is not None and parses
      lenient_extract_ok   = lenient_pred_sql is not None and parses
      first_sql_exec_ok    = execution_errors[0] == "" if all_sql_actions else False
      final_sql_exec_ok    = re-execute(strict OR lenient) succeeds
      final_correct_strict = exec_match(strict_pred_sql, gold_sql)
      final_correct_lenient= exec_match(lenient_pred_sql, gold_sql)
      sql_ast_changed      = canonical_ast(strict_pred) != canonical_ast(first_sql_action)
      table_set_changed    = table_set(strict_pred) != table_set(first_sql_action)
      condition_set_changed= where_clause_set(strict_pred) != where_clause_set(first_sql_action)
      observation_repair_flag = (not first_sql_exec_ok) and final_sql_exec_ok and final_correct_strict
      equivalence_class_change_flag = (canonical_ast(SFT_strict) != canonical_ast(RL_strict))

    Bucket precedence (for {SFT vs RL} pair classification):
      protocol_gain = (not SFT.strict_extract_ok or not SFT.final_sql_exec_ok)
                       AND RL.final_correct_strict
                       AND (RL.lenient_correct == SFT.lenient_correct)
      semantic_gain = SFT.lenient_correct == False AND RL.lenient_correct == True
      no_change     = otherwise
    """
```

**Headline numbers (per RL-vs-SFT pair)**:
- `attribution_ratio = (ΔEX_strict − ΔEX_lenient) / max(ΔEX_strict, ε)` ∈ [0, 1]
- `bucket_share`: % of {SFT-wrong → RL-right} transitions in `protocol_gain` vs `semantic_gain`
- `observation_repair_share`: of `protocol_gain` transitions, fraction with `observation_repair_flag = True`

**Why main novelty**: No published agentic Text-to-SQL RL paper reports either `attribution_ratio` or `bucket_share`. Both are interpretive primitives and generalize to other tool-using agentic settings.

We propose a diagnostic that turns existing aggregate gains into mechanistic claims, and use SKE-RL as designed-to-fail control that makes the diagnostic visible in both directions.

Focused: One classifier + one ratio + one bucket distribution + one supporting failure analysis (predicted by main thesis). No new training for dominant contribution.

Elegant: One re-execution pass + one deterministic classifier + one ratio. Anyone with eval JSONs can apply tomorrow.

**Drift Warning**

**Dominant Contribution**

Sharper. The paper now has one dominant contribution: a deterministic diagnostic decomposition. Collapsing the buckets to `protocol_gain / semantic_gain / no_change` helps. Demoting `observation_repair` to a flag is the right move.

The SKE analysis now reads as supporting evidence rather than a parallel paper.

**Method Simplicity**

Simpler, but not fully clean yet. The method is no longer overbuilt, but the classifier still has one fragile edge: `protocol_gain` depends on strict/lenient correctness in a way that could misclassify records unless lenient candidate selection is fully deterministic and independent of gold execution.

**Frontier Leverage**

Appropriate. This is not old-school. The natural modern primitive here is verifier-style re-execution over agent traces, plus AST canonicalization. No need to add an LLM judge or train another model.

**Remaining Critiques**

1. **Lenient extraction leakage risk**
   - Weakness: “Recover what the agent meant” can become pseudo-oracle evaluation if multiple SQL candidates are extracted and the correct one is selected using execution match.
   - Fix: Define `lenient_pred_sql` as a single candidate selected without gold:
     `final Answer-region SQL if parseable; else last parseable SQL action; else None`.
     Do not choose among candidates by execution correctness.
   - Priority: **CRITICAL**

2. **Classifier condition still underspecified**
   - Weakness: Current `protocol_gain` rule is awkward:
     `RL.lenient_correct == SFT.lenient_correct` does not directly encode “SFT had a recoverable correct SQL but failed strict protocol.”
   - Fix: Use explicit pairwise rules:
     ```python
     protocol_gain = (
         not SFT.final_correct_strict
         and SFT.final_correct_lenient
         and RL.final_correct_strict
     )

     semantic_gain = (
         not SFT.final_correct_lenient
         and RL.final_correct_lenient
     )

     no_change = otherwise
     ```
     Then apply precedence: `protocol_gain > semantic_gain > no_change`.
   - Priority: **CRITICAL**

3. **E3 integration point is weak**
   - Weakness: “SKE class-baseline shifts uncorrelated with semantic_gain records” requires a clear join between training rollout statistics and eval records. Otherwise E3 becomes hand-wavy.
   - Fix: Define the join key now:
     - preferred: exact train/eval record id if same examples;
     - otherwise: query skeleton class / difficulty class / database id;
     - if no reliable join exists, weaken E3 to an aggregate statement about advantage mass by skeleton class.
   - Priority: **IMPORTANT**

4. **Venue readiness still depends on effect size**
   - Weakness: If the +2 pp gain corresponds to very few records, bucket shares may look brittle.
   - Fix: Report binomial confidence intervals or bootstrap CIs over Spider-dev records. This is not extra experimentation; it is required uncertainty accounting.
   - Priority: **IMPORTANT**

**Simplification Opportunities**

1. Drop the optional 3B Dense run from the main proposal. The anchor mentions the 3B cliff, but the paper’s contribution does not need to solve it.

2. Keep the external-baseline probe as a footnote or appendix-only “if artifacts exist” item. Do not let it appear in the main validation plan.

3. Remove M1-M4 from the proposal body entirely. “Appendix only” is fine, but even mentioning it in the core proposal creates latent contribution sprawl.

**Modernization Opportunities**

NONE. The revised method uses the right level of foundation-model-era machinery. Adding an LLM judge would make the diagnostic less clean unless used only for a small human-audit comparison.

**Remaining Action Items**

1. Finalize leakage-safe `lenient_pred_sql` selection.

2. Rewrite bucket rules using direct strict/lenient correctness conditions.

3. Specify the SKE E3 join key or downgrade E3 to aggregate evidence.

4. Add uncertainty intervals for `attribution_ratio` and bucket shares.

5. Keep the main paper centered on C1; make C2 visibly supporting, not co-equal.
