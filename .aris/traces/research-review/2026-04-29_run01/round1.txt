# Research Review Briefing — Multi-turn Text-to-SQL RL: Capacity vs Credit-Assignment

You are a senior ML reviewer (NeurIPS / ICML / ICLR level). Be brutally honest. The author has invested ~3 weeks of GPU time and is deciding whether to write up the work and where to submit. They want a hard, calibrated review now, not encouragement.

## 1. Project history (one paragraph)

Built ReAct-format multi-turn agent for Text-to-SQL on Spider/BIRD using Qwen2.5 (3B, 7B). Pipeline: SFT on teacher-generated ReAct trajectories → GRPO with G rollouts per prompt, multi-turn (Action: execute_sql["..."] → real SQLite execution → Observation injected → continue). Tried 4 reward variants on 3B (dense, sparse, exec_only, CGFR with correctness-gated format scaling) — all flat or negative vs 3B SFT. Reviewer-driven idea search (4 rounds) landed on **SKE-RL**: skeleton-equivalence-class advantage estimation. Hypothesis: the GRPO group baseline is noisy because rollouts with structurally different SQL (different skeleton) shouldn't share advantage; classifying rollouts into canonical AST equivalence classes (`σ_strict` typed slots, `σ_loose` collapsed numerics) and using a size-weighted class baseline should improve credit assignment. Implemented v1 (advantage-side only), with full reviewer iteration (R1-R7) addressing pseudo-class gates, kill-switch ordering, raw-reward RVDS, BIRD layout, etc.

## 2. Final empirical results (full Spider-dev, 1034 examples, single-pass eval at temperature 0)

|---|---|---|---:|---:|---:|

Pre-training gates passed: skeleton extractor 100% coverage on Spider train/dev (947/481 distinct strict classes), within-class variance ≤0.15 median, skeleton diversity ≥40% groups have ≥2 distinct classes.

Auxiliary infrastructure built (all functional, all tested): canonical AST extractor (290 LOC + 98 pytests), class-aware advantage with min-classes / max-extractor-failure-rate / kill-switch gates, skeleton-reward baseline (similarity bonus + standard advantage), DAPO asymmetric clipping, BIRD loader with nested-layout resolver and evidence-aware prompt threading, query-class tagger, checkpoint aggregator with verdict gates, dynamic-padding helper. ~10K LOC, 137 pytests green.
