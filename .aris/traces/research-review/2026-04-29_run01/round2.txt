## F3. Formatting vs Reasoning

**Cheapest distinguishing experiment**

Use existing eval JSONs.

Build a win/loss transition table:

- `format failure`: no valid final SQL/action/final answer.
- `tool failure`: malformed action, failed execution syntax, exceeded turns.
- `SQL executable but wrong`.
- `SQL non-executable`.
- `observation repair`: first attempted SQL wrong/error, later final SQL correct.
- `semantic repair`: final SQL changes skeleton/tables/conditions after observation.
- `no-repair correct`: model gets final SQL right without meaningful tool use.

Then compare SFT vs Dense on SFT-wrong/Dense-right examples.

Also run lenient extraction:
- Strict metric: current EX_all.
- Lenient metric: extract any plausible final SQL from transcript and execute.
- If Dense gain disappears under lenient extraction, it was mostly protocol/format.
- If Dense gain remains on executable SQL and hard query classes, it is semantic.

Cost: near-zero if logs exist; one eval pass per model if not.

**If Dense mostly fixes format/tool compliance**

More interesting for ACL/EMNLP, less interesting for NeurIPS.

It becomes a better paper because the claim is sharper and less inflated:

> Apparent reasoning gains from agentic RL may be largely interface-alignment gains: models learn when and how to call tools, finalize answers, and avoid invalid trajectories.

**Framing if true**

Title-level thesis:

> Agentic GRPO Improves Text-to-SQL by Aligning Tool-Use Protocols More Than SQL Reasoning

Core argument:
- Dense GRPO improves EX_all mainly by reducing invalid trajectories.
- EX_att or lenient SQL execution improves less.
- SKE fails because it targets SQL structure after valid SQL exists, while the bottleneck is earlier: producing valid, executable, well-formed agent trajectories.
- Larger model helps because it has enough base competence to exploit protocol rewards.

That is clean and publishable as a diagnostic paper.

## F4. Can SKE Negative Result Lead?

Currently: no. It is supporting evidence.

As lead, it needs mechanism. “We tried skeleton baselines and they lost” is not enough.

To make it lead-worthy, show one of these:

1. **Variance-reduction failure**
   - SKE does not reduce advantage variance enough to matter.
   - Or it reduces variance but also shrinks useful learning signal.

2. **Class-fragmentation mechanism**
   - Many classes are singleton/small even at G=8.
   - Effective baseline size remains poor.
   - More G increases low-quality off-policy-ish diversity rather than useful within-class comparison.

3. **Wrong abstraction mechanism**
   - Same skeleton often contains semantically different difficulty due to schema linking/literals.
   - Cross-skeleton comparison is actually informative because correct/simple skeletons should beat wrong/complex skeletons.

4. **Optimization mechanism**
   - SKE changes advantage scale/sign distribution.
   - More rollouts produce more negative/near-zero advantages for exploratory but useful trajectories.
   - KL/entropy/reward curves show collapse or under-updating.

Negative result with mechanism:
> “Skeleton-conditioned baselines fail because SQL skeleton is not a sufficient control variate for Text-to-SQL reward: the main variance comes from schema grounding, literal selection, and protocol validity, not AST shape.”

Negative result without mechanism:
> “We tried an idea and it did not work.”

Only the first can lead.

## F5. Competition

I checked current papers because this area moved fast in 2025-2026.

**Direct competitors**

- [ReEx-SQL](https://arxiv.org/abs/2505.12768): execution-aware RL with intermediate database feedback; reports 7B gains on Spider/BIRD. This directly competes with your “multi-turn execution feedback helps” angle. Your differentiation must be diagnostics/negative controls, not headline accuracy.

- [MTIR-SQL](https://arxiv.org/abs/2510.25510): multi-turn tool-integrated RL, GRPO modifications, trajectory filtering, no KL; reports strong 4B Spider/BIRD numbers. This overlaps heavily with your ReAct-GRPO setup.

- [MARS-SQL](https://arxiv.org/abs/2511.01008): multi-agent ReAct-style Think-Act-Observe generator trained with multi-turn RL plus validation agent; reports high BIRD/Spider. This makes a plain “we built multi-turn Text-to-SQL RL” paper noncompetitive.

- [SQL-Trail](https://arxiv.org/abs/2601.17699): multi-turn RL with interleaved feedback, adaptive turn budget, 7B/14B claims. This is dangerous for Story A because it already occupies the “agentic multi-turn RL works at scale” lane.

**Reward / structure competitors**

- [Graph-Reward-SQL](https://aclanthology.org/2025.findings-emnlp.694/): graph-matching and stepwise reward for Text-to-SQL RL; explicitly targets reward quality and SQL equivalence. Your SKE must be positioned as advantage-side, not reward-side.

- [HES-SQL](https://arxiv.org/abs/2510.08896): GRPO plus skeleton-completeness reward and latency reward. This directly competes with skeleton-based structural guidance. Your differentiation: skeleton as baseline/control variate fails, not skeleton reward helps.

- [LearNAT](https://arxiv.org/abs/2504.02327): AST-guided decomposition and margin-aware RL/DPO. Adjacent structure-aware RL; not the same as equivalence-class advantage, but reviewers will cite it.

**Other relevant RL Text-to-SQL**

- [CSC-SQL](https://aclanthology.org/2025.findings-ijcnlp.91/): GRPO for self-consistency/self-correction; strong BIRD private numbers.
- [ExeSQL](https://aclanthology.org/2025.findings-emnlp.1320/): execution-driven bootstrapping and preference training for SQL dialects.
- [Sparks of Tabular Reasoning](https://aclanthology.org/2025.trl-1.20/): GRPO over Text-to-SQL as table reasoning; adjacent, workshop-level.

**Implication**

You cannot win on SOTA. You can win only on:
- reproducible diagnostics,
- negative result with mechanism,
- showing aggregate EX hides protocol-vs-reasoning effects,
- tested infrastructure.
