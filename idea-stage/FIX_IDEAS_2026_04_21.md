# Fix Ideas for Multi-Turn GRPO Reward Failures

## Motivation

Our paper currently has the diagnostic half: we show both dense rewards (variance collapse → template-lock) and sparse rewards (bimodal gradient → early peak + drift) fail in multi-turn ReAct Text-to-SQL agents. To make the story complete and more impactful, we need to **propose and validate fixes**. This converts a "here's why things break" paper into a "here's why + here's how to fix it" paper — substantially stronger for Findings.

**Target paper arc:**
1. Multi-turn GRPO underperforms SFT at 3B → "why?"
2. Dense reward → variance collapse (diagnostic metric: gap > 0.5, var < 0.2)  
3. Sparse reward → bimodal gradient → corr peaks early, drifts  
4. **→ Proposed fixes → empirical validation → at least one fix prevents the failure mode**
5. Practitioner guide: when to use which fix

---

## Literature Gap Map

Key observation: **ALL existing fixes (GDPO, DAPO, Posterior-GRPO, PAPO, VCRL, MO-GRPO) have been validated only on single-turn generation tasks** (math, code, single-turn SQL). None have been applied to multi-turn ReAct agents with real SQL execution feedback. This is our differentiated territory.

## Ranked Ideas

### 🏆 Idea 1: Correctness-Gated Format Reward (CGFR)

**Core hypothesis**: Format/validity/structure rewards should only be awarded when the trajectory ultimately produces correct SQL. This eliminates the "safe floor" that dense rewards create without removing format supervision entirely.

**Mechanism**: 
```python
# In reward_breakdown(): 
if not pred_correct:  # corr = 0
    weights.format *= 0.1   # or 0: penalize format when wrong
    weights.validity *= 0.1
    weights.structure *= 0.1
```
Or the soft variant: `format_weight = base_format × sigmoid(10 × corr)` — smoothly transitions from near-zero to full format reward as correctness improves.

**Relation to prior work**: Posterior-GRPO (2508.05170) and PAPO (2603.26535) gate process rewards on outcome success. We adapt this to multi-turn ReAct format rewards in SQL agents — the mechanism is different (their process reward is a learned PRM; ours is a rule-based format score).

**Novelty**: 7/10 — adaptation of posterior-gating to rule-based multi-turn format rewards. Novel in the SQL ReAct agent context.

**Implementation effort**: 0.5 days — ~15 lines in `reward.py::reward_breakdown()`

**Pilot cost**: 4 GPU-hours (100 steps, G=8)

**Expected outcome**: Dense hacking signature (var < 0.2 at step 175) eliminated or significantly delayed. Correctness curve stays higher for longer.

**Risk**: LOW — if it works, great. If corr still drifts, the pilot still generates data for the "even gating doesn't fully fix multi-turn" narrative.

**Reviewer objection**: "This is just Posterior-GRPO applied to SQL." Counter: We're the first to apply it to multi-turn ReAct agents with multi-component rule-based rewards, and we show WHY it's necessary in this setting (the format structure is richer in ReAct than in single-turn code).

### 🏆 Idea 2: Rollout-Variance Dynamic Sampling (RVDS)

**Core hypothesis**: Skip training steps where all G=8 rollouts have near-identical reward (variance collapse = no gradient signal). This addresses BOTH failure modes: dense hacking (when model locks to template → all 8 get 0.62) and sparse instability (when all 8 fail on hard queries → all get 0).

**Mechanism**:
```python
# In compute_grpo_loss() or the training loop:
reward_var = torch.var(torch.tensor(rewards))
if reward_var < VARIANCE_THRESHOLD:  # e.g., 0.02
    continue  # skip this batch, don't backprop
```

**Relation to prior work**: DAPO (2503.14476) introduces "dynamic sampling" for the same purpose (filter all-correct/all-wrong groups). We extend this to the continuous reward case (not just binary) and apply to multi-turn SQL agents. VCRL does curriculum-based variance filtering but doesn't skip; we skip.

**Novelty**: 6/10 — DAPO already does this for binary rewards. We apply it to continuous multi-component rewards in multi-turn SQL. The novel angle is that our variance threshold catches BOTH failure modes with one criterion.

**Implementation effort**: 0.5 days — ~20 lines in `train_grpo.py`

**Pilot cost**: 4 GPU-hours

**Expected outcome**: No variance collapse signature. Training may take more steps to complete 100 usable updates, but each update carries meaningful gradient.

**Risk**: LOW-MEDIUM — could slow training significantly if many batches are skipped on hard queries.

**Reviewer objection**: "This is DAPO's dynamic sampling." Counter: We apply it to continuous multi-component rewards (not binary 0/1) and to multi-turn agent trajectories, where the variance collapse has a qualitatively different cause (format-floor exploitation vs. all-fail).

### 🥇 Idea 3: Multiplicative Reward Composition (MRC)

**Core hypothesis**: Additive reward composition creates an exploitable floor (format alone can score ~0.69). Multiplicative composition makes correctness a **necessary condition** — the model cannot get meaningful reward without SQL correctness.

**Mechanism**:
```python
# Replace additive weighted sum with:
# total = correctness × (1 + format_bonus × w_f) × (1 + validity_bonus × w_v)
# where format_bonus = max(0, format_score), etc.

total = r_correct * (1.0 + weights.format * max(r_fmt, 0)) \
                 * (1.0 + weights.validity * r_valid) \
                 * (1.0 + weights.structure * r_struct)
```
When `r_correct = 0`: total = 0 regardless of format. The format signals become "bonuses on top of correctness" rather than standalone reward components.

**Novelty**: 8/10 — multiplicative reward composition is a known design pattern (used in some game RL), but has not been applied to Text-to-SQL GRPO or multi-turn agent reward design. The specific formulation (correctness as gate × format as multiplier) is novel.

**Implementation effort**: 0.5 days — ~10 lines in `reward.py::reward_breakdown()`

**Pilot cost**: 4 GPU-hours

**Expected outcome**: Dense hacking eliminated. Floor = 0 when corr=0. Model either learns correct SQL or gets zero reward — binary signal similar to sparse, but with format bonuses when correct.

**Risk**: MEDIUM — effectively becomes sparse reward (corr=0 → reward=0 on most queries), which we know has bimodal gradient problems. May combine with RVDS (Idea 2) to fix that.

**Reviewer objection**: "You've just reinvented sparse reward." Counter: Not quite — when corr > 0, format still provides a gradient bonus (dense-within-correct). And format reward on correct outputs guides the model toward better-structured correct SQL, unlike pure sparse reward.

**Best combination**: MRC + RVDS = correctness-gated dense reward + skip degenerate batches.

### Idea 4: Staged Reward Curriculum (SRC)

**Core hypothesis**: Start sparse (establish correctness), then add dense signals only after a correctness threshold is met. This avoids format-gaming during early training and avoids bimodal gradient on a clean policy.

**Schedule**:
```
Steps 0-25:   sparse only (correctness=1.0, rest=0)
Steps 25-50:  add validity (0.2), keep correctness (0.8)
Steps 50-100: add format+structure, use full dense profile
```
Or adaptive: trigger phase transition when rolling corr > 0.3 for 5 consecutive windows.

**Novelty**: 7/10 — staged reward curriculum is used in robotics RL but not in Text-to-SQL GRPO. No prior paper does staged dense-reward introduction for SQL agents.

**Implementation effort**: 1.5 days — reward profile as a function of training step

**Pilot cost**: 8 GPU-hours (need to run full 100 steps to see the transition effects)

**Risk**: MEDIUM — phase transition timing is a hyperparameter; wrong timing → fall back to one of the original failure modes.

**Why include**: If sparse reward peaks at step 20 (as our data shows), the transition should happen right after the peak. Our existing training curves provide the right timing.

---

### Idea 5: Per-Reward Independent Normalization (GDPO-SQL)

**Core hypothesis**: Instead of normalizing the total reward across G=8 rollouts, normalize each reward component independently. This prevents format rewards (low variance once learned) from dominating gradient magnitude over correctness rewards (high variance, still uncertain).

**Mechanism** (adapting GDPO to our setting):
```python
# Standard GRPO:
advantages = (rewards - mean(rewards)) / (std(rewards) + eps)

# GDPO-style:
adv_fmt = (r_fmt_vec - mean(r_fmt_vec)) / (std(r_fmt_vec) + eps)
adv_corr = (r_corr_vec - mean(r_corr_vec)) / (std(r_corr_vec) + eps)
advantages = weights.format * adv_fmt + weights.correctness * adv_corr + ...
```

**Novelty**: 5/10 — GDPO (2601.05242, NVIDIA NVLabs) is already published. Our contribution is the first application to multi-turn SQL ReAct agents and the insight that the format-correctness variance gap is especially severe in ReAct (because format is solved faster in multi-step trajectories).

**Implementation effort**: 2 days — requires modifying advantage computation in `train_grpo.py`

**Pilot cost**: 4 GPU-hours

**Risk**: LOW technically, but HIGH on novelty — reviewers may view this as "apply GDPO to SQL" without sufficient novelty.

**Verdict**: Include as a baseline comparison (run GDPO-SQL, show it helps but less than MRC or CGFR), not as a primary contribution.

## Recommended Fix Package for Paper

For the paper, we propose and validate 2-3 complementary fixes that together address both failure modes:

### Primary Fix: Correctness-Gated Format Reward (CGFR)
- Addresses: Dense hacking (format floor)
- Cost: 0.5 days code + 4h pilot
- Expected gain: Dense training stays stable 2x longer

### Secondary Fix: Rollout-Variance Dynamic Sampling (RVDS)  
- Addresses: Both (variance collapse detection + sparse instability)
- Cost: 0.5 days code + 4h pilot
- Expected gain: Eliminates degenerate batches from both failure modes

### Ablation: CGFR + RVDS combined
- Does the combination fix BOTH failure modes?
- Expected: Yes — CGFR prevents format floor; RVDS prevents remaining sparse instability
- Total additional compute: 8-12 GPU-hours for 3 ablation runs

## Eliminated Ideas

| Idea | Reason |
|---|---|
| Multiplicative reward (MRC) | Effectively becomes sparse (corr=0→reward=0); RVDS needed to handle that; complex to explain |
| GDPO-style per-reward normalization | Already published by NVIDIA; insufficient novelty for primary contribution |
| Fractional partial-match reward | PaVeRL-SQL already uses this for single-turn SQL; limited novelty |
| VCRL-style variance curriculum | VCRL already published; we can cite but not claim as contribution |
| Staged reward curriculum | High hyperparameter sensitivity; pilot would take 1 GPU-week to tune |
| Entropy exploration bonus | Too many knobs; secondary to addressing the core reward floor issue |
