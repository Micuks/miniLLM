## 2. Setup

### 2.1 Base model and SFT

We start from Qwen2.5-7B-Instruct [cite Qwen2.5] and supervise-fine-tune
(SFT) on a ReAct-format trajectory dataset of 2,000 Spider-train
examples generated by a teacher model under the prompt template
`build_react_inference_prompt(schema, question)`. Trajectories include a
balanced mixture of 70 % single-attempt-success and 30 % error-recovery
examples (one initial Action that fails, one Observation, a corrected
second Action, then Answer). The error-recovery oversampling preserves
the teacher's repair behavior in the SFT distribution.

The SFT recipe uses LoRA-rank-32 adapters over the bf16 base, DeepSpeed
ZeRO-2, gradient accumulation 8 per device, max sequence length 2048,
500 steps. We use the resulting adapter directly as the GRPO starting
point — no merge step is required because QLoRA-GRPO continues training
the same LoRA weights.

### 2.2 Spider-dev evaluation harness

Evaluation runs the policy in interactive ReAct mode on the full
Spider-dev split (1,034 examples), executing each `Action: execute_sql[...]`
against the actual Spider SQLite database file and feeding the result
back as `Observation:`. Per-record metrics:

- `pred_sql`: `extract_final_sql(trajectory)` — the model's final answer.
- `gold_sql`: ground truth from the Spider release.
- `execution_match`: `True` iff the sorted multiset of rows from
  executing `pred_sql` against the real DB equals that of `gold_sql`.
  We use the real DB rather than the schema DDL because some Spider
  questions exercise data values, not just schema shape (per
  `feedback_spider_real_db.md` from earlier project memory).
- `num_turns`: count of model-generated text segments (each ending in
  `Action:` or `Answer:`) in the trajectory.

The harness uses vLLM for generation under all conditions (`gpu_util=0.85`)
to keep evaluation deterministic and fast. The bf16 base + adapter is
merged temporarily for vLLM loading; the merged checkpoint is discarded
after eval.

### 2.3 Reinforcement learning configurations

We compare four GRPO configurations against the SFT baseline. All
configurations share:

- bnb4 (NF4) quantized base + bf16 LoRA-r32 adapter (QLoRA-GRPO);
- 100 rollout-update steps; checkpoint saved every 25 steps; we report
  metrics on `checkpoint-75` (chosen ex ante as the late-but-pre-collapse
  checkpoint, before observing eval numbers);
- max 5 turns per rollout; max 320 generated tokens per turn;
- temperature anneals 0.7 → 0.3 over training; top-p 0.95;
- learning rate 3e-6 cosine with 5 % warmup, KL coefficient 0.1;
- single PPO epoch per group (we verified `old_log_prob = new_log_prob`
  numerically at the on-policy point and skip the redundant actor
  forward; see `feedback_skip_old_logprob.md`);
- the OOM-safe rollout context cap (4096 prompt+gen tokens, observation
  truncated to 120 chars/cell × 2048 chars total, latest-action
  fallback for tokenization budget; see `project_oom_root_cause_fix.md`).

The four configurations differ in `num_generations` (group size G) and
advantage estimator:

| Configuration | G | Advantage estimator |
|---|---:|---|
| Dense G=4 (existing baseline) | 4 | group-relative (`(r − r̄) / σ`) |
| Dense G=8 (B4, this work) | 8 | group-relative |
| SKE-RL G=4 | 4 | class-aware: skeleton-equivalence-class baseline replaces group mean |
| SKE-RL G=8 | 8 | class-aware (same as G=4 SKE-RL) |

The Dense G=4, SKE-RL G=4, and SKE-RL G=8 runs were trained as part of
the v2 SKE-RL investigation; full details and the negative results are
documented in `project_full_spider_dev_dense_vs_ske_g4.md`,
`project_full_spider_dev_ske_g8.md`, and `project_ske_g8_retest_dead.md`.
The Dense G=8 run was added in this work specifically as the B4 confound
test of the v3 pre-registered prediction (see `B4_RESULT_v3_PREDICTION_FALSIFIED_2026-04-30.md`).

### 2.4 Hardware and reproducibility

Single Quadro RTX 6000 (24 GB, Turing sm_7.5, no FlashAttention-2).
DeepSpeed ZeRO-2 with `expandable_segments:True` for VRAM allocator. All
training runs used `seed=42` for prompt-curriculum RNG; CUDA
non-determinism is allowed (the paired-bootstrap CIs absorb the
remaining stochasticity).

Total compute spent on the four GRPO conditions: ~13 GPU-h (Dense G=8)
+ existing artifacts. Eval per checkpoint: ~3 h on full Spider-dev. All
artifacts (eval JSONs, training checkpoints, decomposition outputs) are
named and SHA256-pinned in `PRE_REGISTRATION.md` §1.

### 2.5 Decomposition pipeline

The two-term decomposition is implemented in `miniLLM/diag/` and run via
`scripts/diag_turn_progression.py` (aggregate),
`scripts/diag_turn_by_difficulty.py` (per-difficulty stratification), and
`scripts/diag_paired_turn_transitions.py` (per-record transition matrix).
Bootstrap CIs use `numpy.random.default_rng(seed=20260429)`, 1,000
paired resamples, percentile-95. The pipeline reads only existing eval
JSONs — no GPU is required for decomposition itself. End-to-end runtime
on a CPU is < 10 minutes per pair.

A 32-test unit-test suite (`tests/test_diag_*.py`) covers the lenient
extractor (8 tests, used as a v3-deprecated appendix path),
classifier (10), bootstrap (6), Cliff's δ equivalence (8). The pipeline
is open-sourced under the same license as the rest of the project.
